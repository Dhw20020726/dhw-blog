# Spring Boot + gRPC 微服务全链路 & 面试问答

> 项目：住信SBI网络银行（SSNB NEOBANK）
> 架构：Spring Boot 2.7 + gRPC 1.47 + MyBatis + PostgreSQL
> 模式：BFF（Backend For Frontend）+ gRPC 微服务

---

## 一、一次 API 请求的全链路执行流程

以 `POST /api/account-opening/check-username` `{"username":"testuser"}` 为例：

### Spring Boot 自动配置阶段（启动时）

1. **@SpringBootApplication 启动**
   - @EnableAutoConfiguration -> 自动配置 Tomcat（内嵌）、Jackson、gRPC Client/Server
   - @ComponentScan -> 扫描所有 @Component/@Service/@RestController 注册 Bean
   - 关键自动配置：
     - TomcatServletWebServerFactory -> 启动内嵌 Tomcat (:8080)
     - DispatcherServlet -> Spring MVC 总入口
     - GrpcServerFactory -> Netty gRPC Server (:50051)
     - GrpcChannelFactory -> gRPC Client Channel
     - MethodValidationPostProcessor -> @Validated Bean 的 AOP 校验代理
     - MethodValidationInterceptor -> 织入 @Valid 参数校验

2. **我们自定义的 Bean 注册**
   - RequestContextFilter (@Component + @Order(1)) -> Filter 链第一位
   - ControllerLogAspect (@Aspect + @Component) -> 环绕通知织入 Controller
   - GrpcServiceLogAspect (@Aspect + @Component) -> 环绕通知织入 @GrpcService
   - PgIdInterceptor -> gRPC Server Interceptor
   - AuditInterceptor -> MyBatis Interceptor
   - @GrpcClient("customer-grpc") -> 注入 CustomerServiceBlockingStub
   - ProtobufJsonConverter -> 注册为 HttpMessageConverter

---

### 第一层：Tomcat + Filter 链（Servlet 规范）

```
HTTP 请求到达 -> Tomcat Acceptor 线程接收

Filter1: RequestContextFilter (@Order=1)
  |-- 提取 Header: X-Program-Id, X-Session-Id, User-Agent
  |-- 创建 RequestContext(sessionId, pgId, userId, userAgent, timestamp)
  |-- RequestContextHolder.set(ctx)              [ThreadLocal 线程隔离]
  |-- MDC.put("pgId", ctx.getPgId())            [SLF4J MDC 上下文追踪]
  |-- filterChain.doFilter() -> 放行

Filter2: FormContentFilter (Spring 内置)
Filter3: CharacterEncodingFilter (Spring 内置)
```

### 第二层：DispatcherServlet + AOP（Spring MVC 核心）

```
DispatcherServlet.service()                     [Spring MVC 核心]
  |-- HandlerMapping -> 找到 AccountOpeningController.checkUsername()
  |-- HandlerAdapter -> 准备调用

  ProtobufJsonConverter                         [HttpMessageConverter]
    |-- HTTP JSON -> Jackson -> Map
    |-- Proto Builder: newBuilder() -> setUsername()
    |-- builder.build() -> CheckUsernameRequestModel

  ControllerLogAspect (@Around)                 [Spring AOP 环绕通知]
    |-- [BFF >>>] 检查用户名可用性 请求: CheckUsernameRequestModel{username=testuser}
    |-- joinPoint.proceed() -> 执行 Controller
    |-- [BFF <<<] 检查用户名可用性 完成: CheckUsernameResponseModel{result=taken}

  AccountOpeningController.checkUsername()      [@RestController]
    |-- RequestContextHolder.get() -> 获取线程上下文
    |-- checkUsernameUseCase.execute(request, ctx)
```

### 第三层：BFF UseCase + gRPC Client（BFF 聚合）

```
DefaultCheckUsernameUseCase.execute()           [构造器注入]
  |-- GrpcClientUtil.toProto(request, builder)  [反射 Model->Proto]
  |     Model.username -> Proto.Builder.setUsername()
  |-- GrpcClientUtil.attachContext(stub, ctx)   [@GrpcClient 注入的 Stub]
  |     RequestContext -> gRPC Metadata:
  |       x-session-id, x-program-id, x-user-id, user-agent
  |-- stub.checkUsername(grpcReq)
        |
        v HTTP/2（Protobuf）-> Netty gRPC Server (:50051)
```

### 第四层：gRPC Server + AOP（微服务层）

```
PgIdInterceptor (gRPC ServerInterceptor)
  |-- 从 Metadata 恢复 RequestContext
  |-- gRPC Context.fork() -> Handler 线程恢复

GrpcServiceLogAspect (@Around)
  |-- [gRPC >>>] 检查用户名可用性 请求: username: "testuser"

CustomerGrpcService.checkUsername()            [@GrpcService]
  |-- GrpcServiceUtil.handleUnary(请求, observer, Stub->Model, UseCase, Model->Stub)
        |-- PgIdInterceptor.restore()
        |     RequestContextHolder.set(ctx)
        |     MDC.put("pgId", ctx.getPgId())
        |-- UseCase 调用
```

### 第五层：gRPC UseCase + @Valid 校验 + MyBatis（业务逻辑）

```
DefaultCheckUsernameUseCase.execute()     [@Validated + @Valid]
  | CGLIB 代理 -> MethodValidationInterceptor  [Spring AOP]
  |     |-- Hibernate Validator 校验 @NotBlank @Size @Pattern
  |           失败 -> ConstraintViolationException -> INVALID_ARGUMENT
  |
  |-- customerMapper.existsByUsername(username)

CustomerMapper (MyBatis @Mapper)
  |-- SELECT COUNT(1) > 0 FROM customers WHERE username = ?
        |
        v PostgreSQL -> 返回 boolean
```

### 响应原路返回

```
DB -> Mapper -> UseCase -> GrpcServiceUtil
  -> Model->Stub -> observer.onNext(stubResp) -> gRPC 响应
  -> HTTP/2 -> BFF 收到 -> DefaultUseCase -> mapResponse -> Controller
  -> DispatcherServlet -> Filter 链（倒序）
  -> RequestContextFilter.finally: clear()    [防止内存泄漏]
  -> HTTP 200 JSON -> 浏览器
```

### 简历技术点对应

| 简历技术点 | 在本项目中的体现 |
|-----------|----------------|
| **Spring Boot 自动配置** | @SpringBootApplication -> 自动配置 Tomcat、gRPC Client/Server、MethodValidationPostProcessor |
| **Spring MVC** | @RestController、@RequestBody、DispatcherServlet、@ExceptionHandler |
| **AOP 切面** | ControllerLogAspect / GrpcServiceLogAspect 环绕通知；MethodValidationInterceptor 方法校验代理；AuditInterceptor MyBatis 拦截器 |
| **BFF 聚合层** | web-bff 聚合 gRPC 调用结果返回前端；GrpcClientUtil 封装 Model-Proto 转换和上下文透传 |

| **AOP 异常拦截** | BFF：@RestControllerAdvice -> StatusRuntimeException -> HTTP 400；gRPC：GrpcServiceUtil.handleUnary() -> ConstraintViolationException -> INVALID_ARGUMENT |
| **ThreadLocal 上下文** | RequestContextHolder 线程级存储；MDC 日志追踪；AuditInterceptor 从 MDC 取 pg_id |
| **Proto 代码生成** | generate_models.py 从 proto 注释生成带 @NotBlank/@Size/@Pattern 的 POJO |

## 二、面试问答 17 题

### 1. BFF 是什么，为什么要有 BFF 层？

BFF（Backend For Frontend），为前端定制的后端。聚合多个微服务数据、裁剪字段、把 Protobuf 转 JSON。前端只需要调一个 API，BFF 背后拼数据。

### 2. gRPC 和 REST 有什么区别？为什么内部用 gRPC？

REST 用 JSON 文本 + HTTP/1.1，一次连接一个请求。gRPC 用 Protobuf 二进制 + HTTP/2，长连接多路复用，序列化快 3-10 倍。对内微服务间调用，性能差异巨大。

### 3. Protobuf 比 JSON 快在哪？

JSON 是纯文本（字段名+值原样传输），体积大。Protobuf 是二进制（字段名换成编号，值压缩编码），体积小 3 倍，解析不用逐字符匹配。

### 4. @Valid 和 @Validated 有什么区别？校验的底层原理是什么？

@Valid 用于方法参数或字段上触发校验。@Validated 用于类上开启 Spring AOP 方法级校验。底层 MethodValidationPostProcessor 给 @Validated Bean 创建 CGLIB 代理，MethodValidationInterceptor 在方法执行前调 Hibernate Validator 校验 @Valid 参数。

### 5. CGLIB 动态代理和 JDK 动态代理的区别？

JDK 代理只能代理接口（newProxyInstance），CGLIB 可以代理类本身（继承生成子类重写方法）。Spring 用 CGLIB 是因为 @Validated 加在实现类上。

### 6. DispatcherServlet 和 Tomcat 的关系？

Tomcat 是 Servlet 容器，DispatcherServlet 是运行在里面的 Spring MVC 前端控制器。Tomcat 收到 HTTP 请求交给 DispatcherServlet，它负责找 Controller、调方法、处理异常。Tomcat 只知道它是 Servlet 接口的实现，不知道内部逻辑。

### 7. HTTP/1.1 和 HTTP/2 的关键区别？

HTTP/1.1 文本协议，一次连接一个请求（队头阻塞）。HTTP/2 二进制协议，一次连接多路复用（多个请求同时进行）。gRPC 必须用 HTTP/2 依赖其多路复用特性。

### 8. BFF 用 @RestControllerAdvice 捕获异常，为什么 gRPC 不能？

BFF 走 Spring MVC（DispatcherServlet 有完整的异常处理链）。gRPC 由 Netty 直接处理网络请求，不走 DispatcherServlet，没有全局异常处理器。必须手动 try-catch 通过 observer.onError(Status.xxx) 返回错误。

### 9. RequestContext 是怎么从 BFF 透传到 gRPC 的？

BFF RequestContextFilter -> RequestContext -> GrpcClientUtil.buildMetadata() -> gRPC Metadata 传过去 -> gRPC Interceptor 恢复。不是重新创建，是恢复 BFF 传过来的。

### 10. @Component 和 @Order 在 Filter 上起什么作用？

@Component 让 Spring 发现并注册 Filter。@Order(1) 控制 Filter 执行顺序，数字越小越先执行。Spring Boot 自动把所有 Filter Bean 排好序注册到 Tomcat 的 Filter 链。

### 11. Spring 自动注入 @GrpcClient 的原理？

启动时扫描 @GrpcClient("xxx") 注解 -> 从 application.yml 读取 grpc.client.xxx.address -> 创建 ManagedChannel -> 反射生成 Stub 实例 -> 注入字段。

### 12. 线程上下文 ThreadLocal 在 gRPC 中为什么有问题？怎么解决？

gRPC Interceptor 和 Handler 跑在不同线程，ThreadLocal 无法跨线程传播。gRPC 提供了自己的 Context 机制（Context.fork() + Contexts.interceptCall()），可以跨线程传播。

### 13. 什么是 AOP 切面？怎么定义一个切面？

AOP 在方法执行前后插入逻辑，不侵入业务代码。@Aspect + @Around("execution(...)") 定义环绕通知，joinPoint.proceed() 执行原方法。适合日志、事务、权限等横切关注点。

### 14. 长连接和短连接的区别？gRPC 用长连接有什么好处？

短连接每次请求新建 TCP 连接（需三次握手）。长连接复用已建立的连接，省去握手开销。gRPC 的 HTTP/2 长连接 + 多路复用，BFF 到微服务百万并发只需几十条连接。

### 15. 微服务高并发场景下比单体好在哪？

单体所有业务一个进程，瓶颈一起扛。微服务可独立扩容（比如开户模块忙只扩 customer-grpc），gRPC 长连接池降低网络开销，Protobuf 减少带宽消耗。百万并发时连接数和内存不是线性增长的。

### 16. MyBatis 拦截器怎么实现审计字段自动填充？

实现 org.apache.ibatis.plugin.Interceptor，@Intercepts 拦截 Executor.update()。@Component 注册为 Bean，MyBatis 自动添加。拦截 INSERT 和 UPDATE，从 MDC 取 pg_id，设置 created_at/updated_at/pg_id。

### 17. MDC 是什么？

Mapped Diagnostic Context，SLF4J 的线程级 Map。MDC.put("pgId", "wpl031601") 后，当前线程所有日志自动带上 [pgId=wpl031601]。用于日志追踪和审计字段传递。

## 三、Spring Boot 基础知识点（两年开发应脱口而出）

### 1. Spring Boot 自动配置原理

Spring Boot 启动时通过 `@EnableAutoConfiguration` 触发，读取 `META-INF/spring.factories` 中注册的所有 `AutoConfiguration` 类。每个自动配置类用 `@ConditionalOnClass`、`@ConditionalOnMissingBean` 等条件注解判断是否生效。

例如：`DataSourceAutoConfiguration` 检测到类路径有 `DataSource` 且用户没自己定义 → 自动创建 DataSource Bean。

**一句话：约定大于配置，classpath 有什么 jar，Spring Boot 就自动配什么。**

### 2. IOC / DI（控制反转 / 依赖注入）

IOC：把对象的创建权从自己 new 反转给 Spring 容器管理。DI：Spring 把依赖自动注入到需要的地方。

三种注入方式：
- 构造器注入（推荐）：`public XxxService(XxxMapper mapper) { this.mapper = mapper; }`
- Setter 注入：`@Autowired public void setMapper(XxxMapper m) { this.mapper = m; }`
- 字段注入（不推荐）：`@Autowired private XxxMapper mapper;`

### 3. Bean 生命周期

```
实例化 → 属性填充 → BeanNameAware → BeanFactoryAware
→ BeanPostProcessor.before → @PostConstruct
→ InitializingBean.afterPropertiesSet → init-method
→ BeanPostProcessor.after → 使用中
→ @PreDestroy → DisposableBean.destroy → 销毁
```

### 4. @Component vs @Service vs @Repository vs @Controller

都是 `@Component` 的语义化别名，功能完全相同。区别只是为了分层阅读：
- `@Controller`：表示这是 MVC 控制器
- `@Service`：表示这是业务逻辑层
- `@Repository`：表示这是数据访问层，额外提供异常翻译（JDBC 异常 → Spring DataAccessException）
- `@Component`：通用组件

### 5. @Configuration 和 @Bean

`@Configuration` 声明一个配置类，里面的 `@Bean` 方法返回的对象会被 Spring 管理。`@Configuration` 类本身也会被 CGLIB 代理，保证 `@Bean` 方法只执行一次（单例）。

```java
@Configuration
public class AppConfig {
    @Bean
    public DataSource dataSource() {
        return new HikariDataSource();
    }
}
```

### 6. ApplicationContext vs BeanFactory

- BeanFactory：最底层的 IoC 容器，懒加载，功能简单
- ApplicationContext：BeanFactory 的子接口，启动时预加载所有单例 Bean，支持国际化、事件发布、AOP 等。**Spring Boot 默认使用 AnnotationConfigApplicationContext**

### 7. Spring Boot 启动流程

```
1. new SpringApplication(主配置类)
2. run() 方法：
   ├── 创建 ApplicationContext
   ├── 准备环境（读取 application.yml）
   ├── 打印 Banner
   ├── 扫描加载所有 AutoConfiguration
   ├── refresh() 容器 → 实例化所有单例 Bean
   └── 启动内嵌 WebServer（Tomcat/Jetty）
```

### 8. application.yml 加载顺序与优先级

```
命令行参数 > application-`profile`.yml > application.yml
```

`spring.profiles.active: local` 会额外加载 `application-local.yml`，后加载的覆盖先加载的同名配置。

### 9. @Value vs @ConfigurationProperties

**`@Value`**：从配置文件里取**一个**值，注入到字段上。支持 SpEL（Spring 表达式语言，可以在 `$` 大括号里写默认值、运算等）。

```java
// 基础用法：读取 server.port，读不到用 8080
@Value("${server.port:8080}")
private int port;

// SpEL 示例：注入一个列表
@Value("${app.allowed-origins}")
private List<String> origins;
```

**@ConfigurationProperties(prefix = "xxx")**：把配置文件里**一组**同前缀的属性，批量映射到一个 POJO 类（POJO = Plain Old Java Object，只有字段 + getter/setter 的普通 Java 类）的所有字段上。支持 JSR-303 校验（JSR-303 = Java Bean Validation 标准规范，即 `@NotBlank`、`@Size` 这些注解）。

```java
// application.yml:
// grpc:
//   client:
//     customer-grpc:
//       address: static://localhost:50051
//       negotiation-type: plaintext

@Component
@ConfigurationProperties(prefix = "grpc.client.customer-grpc")
public class GrpcClientProperties {
    @NotBlank  // JSR-303 校验：不能为空
    private String address;
    private String negotiationType;
    // getter / setter ...
}
```

**区别一句话：** 读一两个值用 `@Value`，读一堆有结构的配置用 `@ConfigurationProperties`（类型安全 + 可校验）。

### 10. @Qualifier 和 @Primary — 解决"同一个接口有多个实现类"

当 Spring 容器里同一个接口有多个 Bean 时，`@Autowired` 不知道该注入哪个，启动直接报错。

```java
// 场景：UserService 有两个实现
@Service
public class UserServiceA implements UserService { }

@Service
public class UserServiceB implements UserService { }

// 报错！Spring 不知道用哪个
@Autowired
private UserService userService;
```

**@Primary**：在其中一个上标记"默认选我"，没有歧义时自动选它：
```java
@Service @Primary
public class UserServiceA implements UserService { }
// 现在 @Autowired UserService → 自动注入 UserServiceA
```

**@Qualifier("beanName")**：注入时明确指定用哪个 Bean 的名字：
```java
@Autowired @Qualifier("userServiceB")
private UserService userService;  // 精确指定用 B
```

### 11. @Lazy — 延迟初始化

Spring 默认启动时把所有单例 Bean 都创建好（饿汉式）。`@Lazy` 让 Bean 在**第一次被用到时才创建**：

```java
@Service
@Lazy
public class HeavyService {
    public HeavyService() {
        // 这个构造器不会在启动时执行，而是在第一次调用时才执行
    }
}
```

**使用场景：** 某个 Bean 初始化特别慢（比如要建立外部连接），而且不是每次请求都用，就加 `@Lazy` 加快启动速度。

### 12. @Scope — Bean 的作用范围

Spring Bean 默认是单例（singleton）：整个容器只有一个实例。`@Scope` 改作用域：

| scope | 含义 | 场景 |
|-------|------|------|
| `singleton`（默认） | 整个容器唯一实例 | 无状态 Service、Mapper |
| `prototype` | 每次获取都 new 一个新实例 | 有状态的、每次请求不同的对象 |
| `request` | 每个 HTTP 请求一个实例 | Web 环境下请求级 Bean |
| `session` | 每个 HTTP 会话一个实例 | 用户登录会话级 Bean |

```java
@Service
@Scope("prototype")
public class OrderProcessor { }  // 每次注入都是新对象
```

### 13. @PostConstruct 和 @PreDestroy — Bean 生命周期钩子

```java
@Service
public class CacheService {

    @PostConstruct  // Bean 创建完成后自动执行
    public void init() {
        // 启动时加载缓存、预热数据
        loadDataIntoCache();
    }

    @PreDestroy  // Bean 销毁前自动执行
    public void cleanup() {
        // 关闭连接、释放资源
        closeConnections();
    }
}
```

**与构造器的区别：** 构造器执行时依赖还没注入完，`@PostConstruct` 执行时所有 `@Autowired` 字段已经注入完毕，可以安全使用。

### 14. @Profile — 环境切换

同一个 Bean 只在指定环境下激活，其他环境下不创建：

```java
// 开发环境用内存数据库
@Configuration
@Profile("local")
public class LocalDataSourceConfig { }

// 生产环境用真实数据库
@Configuration
@Profile("prod")
public class ProdDataSourceConfig { }
```

激活方式：`spring.profiles.active: local`。我们的项目就是通过这个区分 `config/local/`、`config/ita/`、`config/prod/` 三套配置。

### 15. @ConditionalOnClass / @ConditionalOnMissingBean — 条件装配

Spring Boot 自动配置的核心注解。**条件满足才创建 Bean，不满足就跳过。**

```java
// classpath 有 DataSource 这个类时，才创建这个 Bean
@Bean
@ConditionalOnClass(DataSource.class)
public JdbcTemplate jdbcTemplate() { ... }

// 用户没有自己定义 DataSource 时，才自动创建一个默认的
@Bean
@ConditionalOnMissingBean(DataSource.class)
public DataSource defaultDataSource() { ... }
```

这就是为什么你加了 `mybatis-spring-boot-starter` 依赖就自动有了 Mapper，去掉依赖就没了——自动配置类上挂了 `@ConditionalOnClass(SqlSessionFactory.class)`，classpath 有就配，没有就不配。

### 常用注解速查（更新版）

| 注解 | 作用 | 一句话场景 |
|------|------|-----------|
| `@SpringBootApplication` | = @Configuration + @EnableAutoConfiguration + @ComponentScan | Spring Boot 启动类 |
| `@RestController` | = @Controller + @ResponseBody | REST API 控制器 |
| `@RequestMapping("/api/xxx")` | URL 映射到方法 | 定义接口路径 |
| `@PostMapping / @GetMapping` | 限定 HTTP 方法 | POST 新增 / GET 查询 |
| `@RequestBody` | HTTP Body JSON → Java 对象 | 接收前端 JSON |
| `@PathVariable` | URL 路径中的变量 | RESTful 路径参数（如 /user/123） |
| `@RequestParam` | URL 问号后的参数 `?page=1` | 查询参数 |
| `@Autowired` | 自动注入 Bean | 依赖注入 |
| `@Qualifier("name")` | 指定注入某个名字的 Bean | 同接口多实现时指定 |
| `@Primary` | 同类型多 Bean 时默认选我 | 标记默认实现 |
| `@Lazy` | 用到才初始化，不在启动时创建 | 慢 Bean 延迟加载 |
| `@Scope` | Bean 是单例还是每次新建 | singleton/prototype |
| `@PostConstruct` | Bean 创建完毕自动调 | 初始化缓存/连接 |
| `@PreDestroy` | Bean 销毁前自动调 | 释放资源 |
| `@Profile("local")` | 指定环境才激活 | 环境切换 |
| `@ConditionalOnClass` | classpath 有某类才生效 | 自动配置的条件判断 |
